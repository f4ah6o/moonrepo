## 学び

XState の guard は、遷移できるかどうかを純粋な boolean 条件で決める仕組みで、
`domainprocessschema.mbt` の `guard` とかなり近い。ただし XState は同じ event に対して
複数 guard を順に評価して fallback へ進めるが、現行 schema にはその ordered fallback がない。
そのため、今は「guard がすべて false なら available transition なし」としてテストし、
fallback は将来仕様として扱うのが自然だと判断した。

Flowable と Camunda の gateway 例は、BPMN XML そのものではなく、承認可否や金額による
排他的分岐の業務ケースとして使える。BPMN 互換は v1 の非目標なので、XML import/export に
寄せず、YAML fixture に再表現する方が実装に効く。

JSON Forms の rule は、条件が成立した時に hide/show/enable/disable を UI 要素へ適用する。
現行 `domainprocessschema.mbt` では rule は active/inactive として runtime projection に出るだけで、
field visibility を直接変えない。この差分は、今後の GUI manifest 仕様を広げる時の境界として使える。
