## FWD.mbt workbench preview の dslx L1 化

今日は `FWD.mbt` の dslx 再構築を、CLI 入口から workbench preview 側へ進める。前段までで `compile_dslx_l1` と `compile-dslx` / `validate-dslx` は merge 済みなので、次は authoring-first な `workbench_v6` が直接 `Schema` を組み立てるだけでなく、dslx L1 text を生成して `compile_dslx_l1` を通す経路を持つようにする。

変更対象は `target-repos/FWD.mbt.git/.wt/codex/dslx-workbench-preview` の `src/workbench_v6/workbench.mbt` と `src/workbench_v6/workbench_test.mbt` を主にする。必要なら README に短い説明を足すが、既存の YAML CLI、runtime、effects、hypermedia、frozen API は変更しない。

実装計画は小さく保つ。`BuilderState` から L1 DSL 文字列を作る helper を追加し、`evaluate_builder` の IR 生成を `builder -> dslx L1 text -> @compiler.compile_dslx_l1 -> runtime preview` に寄せる。既存の `builder_to_schema` は互換のため残し、テストで生成された dslx L1 が parse/compile できること、default builder と purchase request case の preview が従来どおりであること、invalid transition target がこれまで同様に validation error になることを確認する。

検証は対象 worktree で `moon check` と `moon test` を実行する。MoonBit 作業なので `moonbit-agent-guide` を使い、API 形状や package 境界に触れる場合は `moonbit-refactoring` の「小さく安全な変更」を優先する。

## 実施結果

`workbench_v6` の preview IR 生成を `builder -> dslx L1 text -> compile_dslx_l1 -> runtime preview` に寄せた。`builder_to_schema` は互換の validation 用に残し、dslx L1 が先頭 state を `initial_state` として読む制約に合わせて、生成時は initial state を先頭へ並べ替えるようにした。

opencode review loop の 1 回目では、二重 validation の意図、`Int` sentinel、DSLX 識別子の扱いが指摘された。二重 validation は旧 schema path と同じ semantic reason を返すための互換ガードとしてコメントを残し、sentinel は `Option[Int]` に変更した。DSLX が受ける名前は `[A-Za-z_][A-Za-z0-9_]*` なので、Workbench 側でも非空かつ不正な識別子を `WORKBENCH_FIELD_INVALID` として compile 前に返すようにした。

対象 worktree で `moon fmt src/workbench_v6/workbench.mbt src/workbench_v6/workbench_test.mbt`、`moon check`、`moon test` を実行し、既存 warning は残るが 423 tests passed を確認した。2 回目の opencode review loop は exact `LGTM` を返したため、PR #13 を draft 解除して merge した。merge commit は `a8c1e02`。
