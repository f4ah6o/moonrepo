## dslx.mbt 新規作成計画

今日は `f4ah6o/dslx.mbt` を新規 target repo として作り、MoonBit で DSL 生成用メタフレームワークの動く MVP を実装する。`target-repos/dslx.mbt.git/.wt/main` は baseline checkout として固定し、初期 repo 作成後の実装は新規 repo の初期 commit としてまとめる。

変更対象は moonrepo 側では `repository.ini` とこの日誌だけにする。既存の `docs/2026-05-30.md` と `docs/2026-05-31.md` には未コミット変更があるため、今回の作業では触らない。

`dslx.mbt` 側は Apache-2.0 の MoonBit module として初期化し、`dslx/core`、`dslx/astx`、`dslx/pars`、`dslx/vald`、`dslx/prnt`、`dslx/emit`、`dslx/test` の package を作る。公開 API は 4 文字名に統一し、`dslx`、`node`、`case`、`fldx`、`rule`、`seqx`、`altx`、`many`、`optx`、`litx`、`tokn`、`rgxx`、`prat`、`pref`、`infx`、`post`、`vald`、`chec`、`warn`、`fail`、`hint`、`diag`、`span`、`prnt`、`grup`、`line`、`join`、`nest`、`brkx`、`emit`、`file`、`mods` などを builder として提供する。

MoonBit の `test` は構文語で、この toolchain ではバッククォートによる公開関数名にもできなかったため、Test DSL の入口は 4 文字の `tstx` にする。同様に top-level `init` も特別扱いされるため、state machine の初期状態指定は method `.init(...)` として提供する。

MVP では実ファイルを書き込む codegen は作らない。`emit(...).file(...).mods(...)` は仮想ファイルの path と content を返すだけにし、生成結果をテストしやすくする。検証は `moon check`、`moon test`、`moon info` を実行し、公開 API の `pkg.generated.mbti` を確認する。

## dslx.mbt Core v1 仕様化計画

次の作業では `dslx.mbt` の Core v1 を、仕様と acceptance test を中心に固める。`target-repos/dslx.mbt.git/.wt/main` は baseline として更新し、実装は `just codex-start dslx.mbt core-v1-spec` で作る専用 worktree で行う。

変更対象は target repo 側の `dslx.mbt`、`dslx_test.mbt`、`README.mbt.md`、`pkg.generated.mbti` を中心にする。必要に応じて root package 内でファイル分割してもよいが、API の見え方は package 単位なので file 名を namespace として扱わない。

Core v1 では AST/Grammar/Parser/Validator/Printer/Codegen の契約を README に明文化する。Grammar には named rule reference の `refx` を追加し、spec/rule から開始 rule 名で parse する `runx` を追加する。`rgxx` は silent literal fallback をやめ、対応 pattern を仕様化し、未対応 pattern は validator または parser diagnostic で失敗させる。

Validator は空名、重複 node/case/field/rule、未定義 `refx`、未対応 `rgxx`、空 `seqx`/`altx` を検出する。Codegen は virtual file のまま維持し、`pars.mbt` が rule body の式を使うことを snapshot test で固定する。

検証は `moon fmt`、`moon check`、`moon test`、`moon info` を実行する。仕上げに opencode-review-loop を使い、exact `LGTM` まで指摘を潰す。

## dslx.mbt Parser v2 実用化計画

次の作業では `dslx.mbt` の parser を実用寄りにする。`target-repos/dslx.mbt.git/.wt/main` を `origin/main` へ fast-forward し、merged 済みの `completion-v02` と `core-v1-spec` task だけを整理してから、`just codex-start dslx.mbt parser-v2` で専用 worktree を作る。

変更対象は target repo 側の `dslx.mbt`、`dslx_test.mbt`、`README.mbt.md`、`pkg.generated.mbti` にする。breaking change は許容し、`Pout` に `done : Bool` を追加する。`pars` と `runx` は末尾空白以外を全消費した時だけ成功扱いにし、未消費入力がある場合は `done=false`、`tree=None`、`fail("unexpected input")` の diagnostic を返す。

空白は現状通り token 前で自動 skip する。parse 失敗 diagnostic には必ず `span("", posx, posx)` か該当範囲の span を付ける。`refx` は現行の wrapper tree を維持し、`rgxx` は Core v1 の対応 pattern だけを引き続き許可する。

テストでは完全一致、末尾空白、未消費入力、literal/token/rgxx/refx/start-rule 失敗の span、`refx` wrapper、`many`/`optx`/`altx` の既存診断方針、README の checked example を確認する。検証は `moon fmt`、`moon check`、`moon test`、`moon info`、opencode-review-loop の exact `LGTM` まで行う。

## dslx.mbt multi-DSL project framework v2 実装計画

次の作業では `dslx.mbt` の `Proj` を、複数 DSL spec をまとめて扱う project framework として一段実用化する。`target-repos/dslx.mbt.git/.wt/main` は baseline のまま固定し、実装は `just codex-start dslx.mbt dslx-project-v2` で作る専用 worktree で行う。

変更対象は target repo 側の root package 実装、README、QuickCheck を含むテスト、`pkg.generated.mbti` にする。追加 API は `fspc(prj, name)`、`runp(prj, spec, rule, text)`、`chex(prj, spec, rule, file)` の最小 3 つに絞り、既存の `Proj` field と builder API は壊さない。

`fspc` は `Proj` から spec 名で `Dslx?` を探す。`runp` は project-level parser entrypoint として spec 名と rule 名を受け取り、見つかれば既存 `runx` と同じ結果を返し、見つからない spec は fail diagnostic を含む `Pout` を返す。`chex` は example file の text を `runp` で検証し、成功なら空 diagnostic、失敗なら parser diagnostic を返す。

テストでは `fwds`、`wflw`、`domn`、`n8nx` を 1 つの `Proj` にまとめる smoke test と、QuickCheck による project lookup、`runp` と `runx` の一致、example check、pack order、duplicate diagnostics の不変条件を追加する。検証は対象 worktree で `moon check`、`moon test`、`moon info`、`git diff --check` を実行し、最後に `opencode-review-loop` を exact `LGTM` まで回す。
